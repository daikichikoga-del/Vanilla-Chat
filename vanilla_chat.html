用語集

APIサーバーURL

このチャットがリクエストを送信するサーバーのアドレスです。例：https://api.example.com/v1/chat/completions。通常は OpenAI 互換の形式で、/v1/chat/completions をエンドポイントとして使います。

API キー

サーバーに対して「あなたは誰か」を証明するための認証情報です。秘密鍵（シークレット）として扱い、他人に共有してはいけません。テキスト形式で保存され、リクエストヘッダとして送信されます。

署名認証（HMAC）

APIキーとリクエストの内容から計算されるハッシュ値で、「このリクエストが改ざんされていない」ことを証明します。サーバー側で署名を検証して、正当なリクエストのみを受け入れます。おてもとLLM などが採用しています。

usage（使用量）

APIの回答に含まれる、このリクエストで消費したトークン数の情報です。prompt_tokens（入力）と completion_tokens（出力）に分かれており、従量課金型のAPIでは課金根拠となります。

stream_options.include_usage

ストリーム形式（リアルタイム逐次表示）でAPIから回答を受け取る際に、同時に使用量（usage）情報も含めるオプション。対応していないサーバーでは無視される場合があります。

自動要約（メモリ圧縮）

会話が増えすぎたとき、古い会話を短い文章（要約）に圧縮して、新しい会話とセットで送信する仕組みです。これにより、長い会話でもAPIの送信量を抑えながら文脈を保つことができます。

JSON 保存機能

このチャットの会話履歴と長期メモリを JSON 形式で保存・復元するツール。ブラウザのローカルストレージだけでなく、ファイルとして他のPCやツールで読み込むことができます。

メモリ設定

直近の会話はそのままAPIへ送り、古くなった会話は自動要約で圧縮します。要約は回答後に行うため、通常の返答開始は遅くなりにくい設計です。単発モード中はこの設定は使われません。

おすすめは4往復です。多くすると文脈は残りますが、送信量が増えます。

この数を超えたら、古い会話を長期メモリへ圧縮します。直近保持より大きい値にしてください。

CONTEXT GUIDE  コンテキスト保持とは

API型チャットで「前の話を覚えている」ように見せるための基本的な考え方です。

1. APIのモデルは自動では会話を覚えません

OpenAI、Gemini、その他の多くのLLM APIでは、基本的に1回のAPIリクエストに含まれている情報だけを見て回答します。 ブラウザやアプリの画面に過去の会話が残っていても、それを次のAPIリクエストに含めなければ、モデルは前のやり取りを参照できません。

つまり、APIを使う側のアプリが「今回の質問」と一緒に「必要な過去の会話」を送る必要があります。

今回だけ送る場合:
user: この続きは？

このように送ると、モデルには「この」が何を指すのか分かりません。

2. 定石はローカル側で履歴を保持して毎回送ること

チャットアプリでは、ユーザーのPC、ブラウザ、アプリ、またはサーバー側で会話履歴を保持し、 次のリクエスト時に必要な範囲の履歴を messages として一緒に送るのが一般的です。

毎回送る内容の例:
system: あなたは日本語で回答するアシスタントです
user: Aについて教えて
assistant: Aは...
user: ではBとの違いは？

このように送ると、モデルは「Bとの違い」が「Aとの違い」だと判断しやすくなります。

3. このチャット版の方式

このコンテキスト保持版では、すべてをそのまま送り続けるのではなく、次のようなハイブリッド方式を採用しています。

直近履歴 最近の数往復は、そのままAPIへ送ります。細かい表現や直前の流れを残すためです。
長期メモリ 古い会話は、回答後に短い要約へ圧縮して保存します。次回以降はその要約を送ります。

この方式の狙い

  • 直近の会話の自然さを保つ
  • 古い会話を短く圧縮して送信量を抑える
  • 毎回要約しないことで、通常の返答開始を遅くしにくくする
  • DBを使わず、ブラウザ内だけで完結させる
古い会話の要約は、回答が終わったあとに行います。そのため、通常の質問では先に回答を返し、次回以降のために裏側で会話を整理します。

4. 他にもいろいろな方法があります

コンテキスト保持には、このHTMLで使っている「直近履歴 + 要約メモリ」以外にも複数の方法があります。

直近履歴だけ送る

最も簡単な方法です。古い話は忘れやすいですが、実装が軽く、動作も分かりやすいです。

会話を要約して送る

古い会話を短い要約にして送る方法です。長めの文脈を保ちやすくなりますが、細かい表現や数値が落ちることがあります。

RAGを使う

RAGは、外部の文書、FAQ、社内資料、過去ログなどを検索し、関連する情報だけをAPIリクエストに含める方式です。 大量の情報から必要なものを取り出す用途に向いています。

サーバー側で履歴を管理する

ユーザーごとの会話履歴をDBに保存し、サーバー側で必要な履歴を選んで送る方式です。 本格運用には向いていますが、DB、認証、削除機能、個人情報管理などの設計が必要になります。

5. この版での注意点

  • 会話メモリはこのPCのブラウザに保存されます。
  • 別のPCや別のブラウザには引き継がれません。
  • 共有PCでは、利用後に「会話メモリを削除」を実行してください。
  • 要約メモリは便利ですが、古い会話を完全に保存するものではありません。
  • 重要な条件、数値、ファイル名、URLは、必要に応じて再度チャット内で明記してください。
